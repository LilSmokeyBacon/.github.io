8080 processor


A complete 8080 microprocessor emulator that can run classic space invaders and some COM programs, I mainly chose this emulator to learn more about emulator development and low-level programming.

All 255 opcodes have been implemented and tested, also some dedicated shift hardware has been emulated as space invaders use this for moving pixels on the screen and thus adds a hardware shift register to help with the math.

As the 8080 is a microprocessor and not a complete machine like a Gameboy, it does not have a standard input or sound, only some ports that can receive and send signals. So for sending keyboard signals I used Qt as these are not really part of emulating an 8080. For now there's no sound but emulating a sound card used in old arcade machines might be a fun project for the future.

Reinforcement Learning


Description

This was a research project I did about reinforcement learning in 2D shooter games, while this project didn't turn out as well as I hoped it would, I did learn a lot from it and really enjoyed it.
The code in this project isn't the best as it was mainly focused on research. However besides these things I did get some nice results and managed to write a small article about the process.

You can find the full research paper with all the code on my github (see top left for link)
but here's my conclusion.


Conclusion

The aim of this paper was to see how well I can use RL to learn a bot basic AI behaviours and how well it would perform in a 2d shooter. From my attempt, I can conclude that basic AI behaviours aren't too hard to teach, but teaching an agent how to properly navigate or explore a map is a lot harder. I do believe that it is possible, if you have the required knowledge and time for it. When you do manage to do it properly, I can see various advantages for using Rl over rule-based systems such as; - Minimal code needed for the underlying algorithm in some cases - Decrease in the time spent tuning parameters for different situations - The ability to produce different bot personality types I'd say that teaching an agent how to explore a room was hard for my low experience when starting this project - combined with the limited time. It might have been better to start off simpler. However, I did learn a lot about machine learning in general and I have improved dramatically. So therefore - when I have time - I might restart or continue on this project and try to make something better. Further work could be spent on trying different environmental setups with the same agents to see how well they perform in an unknown environment. Some of the tasks have already been tested in different environmental setups like the searching for items and shooting an enemy as those setups were random. There is a clear future for RL in games, and the potential is limitless. When you combine it with more conventional rule-based systems, the options are endless. I am genuinely excited to witness the progressions.

Raymarching

Description

This was my first time using unity's compute shader so I started off by making a small test project randomizing the color and position of some cubes,
I found that it wasn't too diffrent from a class I had in school where we made a raytracer and rasterizer in C++ using DirectX and HLSL,so it was suprisingly easy to get in to.
After that I felt confident enough to start on making the raymarching project.

Making the actual raymarching shader was suprisingly easy, having made a raytracer in the past, it was interesting to see the differences between
the raytracer and raymarching and seeing whats possible using the signed distance formulas raymarching uses.
If you're interested you can find and download my project and or read more about it on my github.


Result

Using signed distance functions allows you to do some operation between shapes like masking, blending, and cutting and by combining shapes and using those operation you can create many thing's (some more usefull than others) but regardless of the use case it can be very fun to play around with.

The Crunch

Game Jam

I made this game with a good friend for the Ludum Dare 42 game jam. This was before I started studying at DAE and even though I knew a decent amount about C++, the good practices were definitely not there.


Mistakes

There's so much wrong with the codebase that I can't possibly give you focused code snippets. Restarting the game was an afterthought I fixed with a goto, causing the biggest memory leak imaginable. I knew about constructors, but not about the initializer list and didn't use the destructor. So goodbye RAII principles and hello even more memory leaks! The Spaceship's render method wasn't const and has a shocking amount of gameplay logic in there.
The list goes on, really.
(You can find the code via our LDJam page, but I won't link to it directly to prevent people from opening it without reading this.)


Conclusion

I guess the reason this project is still on my portfolio is simply that it was my first "real" game, we did a pretty good game jam even though our skillset was lacking to say the least.
It's also an indicator on how far I've come since then and a reminder of why I got into videogames.

The Ascend


You can check out the whole progress of making the game here


The Ascend is a game where the two main characters Bling and Blong fell down a ravine and on their way out they uncover the mysteries of a long lost tomb.
It's a game where you have to work together to solve puzzles and make your way trough the level, the only catch is that you bound together by a rope, luckly you can use this rope to slove the puzzles and get out of the ravine.
Everything was made in unity and version controlled with perforce.


The team and responsibilities


We were a team of 1 programmer and 3 artists. Initially, we had 2 programmes but one of them dropped out before during the prototyping phase.
So essentially all the programming was done by myselfs.
